# The Net — Design & Implementation Plan

Status: design locked, ready to build. Target: the reserved `web` section of the
`app.html` shell (branch `redesign/aside-browser`). LAN-only, Yjs-synced,
multi-campaign. This document is the single source of truth; it supersedes the
scattered notes in chat and memory.

How to read it: **Part A** is the concept (what we're building and why).
**Part B** is how it maps onto the *existing* codebase (grounded in the real
modules). **Part C** is the phased plan (v1 → v2 → v3). **Part D** is the schema
appendix. **Part E** lists deferred/open items.

**Locked build decisions:**
- **Delivery time** is driven by a **GM-advanced session clock** (hours/days), not
  the coarse `CS.lifestyle.month` counter.
- **v1 purchase model** = **player-side debit + delivery only**; the two-way loop
  (stock decrement + owner-revenue credit) lands in **v2** with the traffic/ads
  economy.
- A **`site` is its own entity**, linked to its subject (`location`/`org`/`shop`/
  `npc`) via the Links registry — never an embedded field on the subject.
- Computer vs deck (E.2): **kept separate**, with reach computed as the max of the
  two.

# PART A — Concept

## A.1 What the Net is

The Net is a second, navigable layer over the campaign world. Every meaningful
thing in the game — a place, an organisation, a shop, a person, an item — can
project a **presence** on the Net. Players reach that presence with a computer;
netrunners go *under* it.

The design rests on one canonical image (Grubb's "flashlight in a dark cave"):
the Net is an infinite dark space; a powered, connected machine casts a **pool of
light** around itself; everything else is unrendered **Wilderspace**. Turn a
machine off and its presence vanishes. This is why the Net map is *generated by
the campaign's entities* rather than authored as a fixed map — and it gives us a
free, diegetic fog-of-war.

## A.2 Three tiers, two seams

A single subject (say, Arasaka) exists at three depths:

1. **Open surface** — the public site. Anyone with a computer can browse it. No
   hacking, no login.
2. **Authenticated platform** — the private corporate intranet/extranet behind a
   `login`. Entered with **credentials** (stolen, borrowed, social-engineered, or
   legitimately held by a Corporate PC) — **not** by netrunning. Blocks are scoped
   by clearance, so one site shows different faces to different logins.
3. **Infrastructure (datafort)** — the deep system, behind the code gate. Reached
   only by **netrunning**. This is where the truly secret data and the ICE live.

Two seams: `login` separates 1→2 (key = credentials); a backend link separates
2→3 (key = a netrun). The netrunner's job is to go *under* the surface — e.g. to
crack the datafort and **forge a high-clearance credential** that the party's
infiltrator then uses to walk the platform (and the building). Two professions,
one run.

A site never advertises "hack me": tier 3 is discovered from the infrastructure
side, never surfaced as a button on the public page.

## A.3 Presence is typed by entity

| Subject entity | Net presence |
|---|---|
| location, org, shop, person (npc) | a **site** (open surface ± authenticated platform ± datafort) |
| item | a **MicroNet** (its object-hacking interface — deferred, see E.1) |

A **site** is its own object that *links* to its subject entity; blocks inside the
site pull live data from that subject (a shop-site's `item`/`price` blocks read
the linked `shop`; a person-site's `person` block reads the linked `npc`).

## A.4 Reach vs broadcast — who can connect to what

Two independent properties decide connectability, on a discrete graph of regions
(the game's existing geography — Night City districts, then national, then
global). No continuous distance metric.

- **Reach (a property of the player's computer):** how far the player can traverse
  the graph from their entry point, and which link-security tiers they can cross.
  A weak terminal reaches only its own district; a better rig reaches adjacent
  districts; a top rig + satellite link reaches national/global. Reach is set by
  the **computer object on the sheet** (see B.4). This is the literal size of your
  flashlight.
- **Broadcast (a property of the site):** how far the site itself lights up —
  `local` (its district only), `district`, `citywide`, `national`, `global`. Big
  players (corps, media, emergency services) pay to broadcast wide, so they are
  reachable from anywhere; a neighbourhood BBS only lights up locally and requires
  proximity.

**Connectable if** the site's broadcast covers the player's position **OR** the
site sits within the player's reachable set. Upgrading the computer literally
widens the pool of light — better hardware captures more sites on the map. A
netrunner (or a pre-established access, see B.7) can also cross links a plain user
cannot. Non-runners are strictly *local + broadcast-reachable*.

## A.5 The two audiences

- **Players** browse the Net **autonomously**, each on their own client, seeing
  only what their computer reaches. To them the Net is a translucent browser
  (address bar + map homepage + directory) that is always in their pocket via a
  comms dock (v2).
- **The GM** authors sites, hosts, and infrastructure, drives Netwatch and AIs,
  monitors comms, and casts netruns to the table.

# PART B — Mapping onto the existing codebase

Everything below is grounded in the current `redesign/aside-browser` architecture.

## B.1 Where it lives — the `web` section

The shell (`js/app-shell.js`) already reserves the module: `SECTIONS` contains
`{key:'web', disabled:true}` ("Coming soon", `renderWebPlaceholder`). Build-out:

1. In `app-shell.js`: flip `disabled` off on `web`, give it real `tools` ids,
   add entries in `TOOL_LABEL` / `GLYPH` / `TOOL_SECTION`, and a `tabUrl` case.
2. New module `js/app-web.js` publishing `window.WebSection` (mirror
   `DataSection` / `CastSection`): `renderHome(t, host)` plus per-tool renderers.
   Register its `<script>` in `app.html` before `app-shell.js` / `app.js`.
3. Wire it into the central dispatcher `renderPane` in `app-shell.js`
   (`else if (t.tool === 'web-home' && window.WebSection) WebSection.renderHome(...)`).
4. Per-campaign UI state (open Net tabs, last address, filters) → `meta.ui` via
   `App.uiGet/uiSet`.

The Net is thus just another browser section, consistent with `party/data/map/cast/combat`.

## B.2 Entities & persistence — reuse the typed Store

New Store types, added to `Store.TYPES` **and** to `DOC_TYPES` in
`hub/createHub.mjs` (so the per-campaign folder is created and the REST API
accepts it):

- `site` — the core Net object (block document + theme + reach/broadcast + host
  ref + access-scoped structure). v1.
- `host` *(optional as a distinct type)* — the user's rule is "a host is first of
  all a site like the others," so model a host as a `site` carrying a `hostConfig`
  (see D.3), not a separate primitive. The **host registry** is a directory view
  over host-sites. v1.
- `datafort` — the infrastructure under a site (nodes, walls, gates, ICE, files).
  v3.

Relations use the existing **Links** registry (`campaigns/:cid/links/_links.json`,
refs `{type,id}`): a `site` links to its subject (`location`/`org`/`shop`/`npc`),
to its `host`, and (backend, GM-only) to its `datafort`. This is the graph that
makes "every thing a site" work without embedding.

Player visibility reuses the system props: a site is visible to players when
`props.public` (or `props.contact`) is set — `Store.visibleToPlayers(json)`
already encodes this, and player sessions already get a read-only Store/Links
layer. Unlisted sites (A.4, question 7) = `public` but flagged `unlisted:true`
(reachable by address, absent from the directory).

## B.3 Live state — reuse Yjs

Real-time blocks (`board`, `chat`, `presence`, live `traffic`, ad impressions) run
on the already-wired Yjs layer (`window.BartmossSync`, built from `src/sync/*.ts`
into `js/sync.bundle.js`). Add a `getMap('net')` to `CampaignStore.ts` /
`Campaign.ts` keyed by `siteId` → `{ board:[], presence:{}, traffic:{} }`, expose
accessors, and rebuild the bundle (`npm run build:sync`). Because every campaign is
its own Yjs room, Net live-state is isolated per campaign automatically.

For GM→player pushes (casting a run, forcing an event) reuse the existing
`overview` map + `castReveal` pattern from `app-cast.js` — no new transport needed.

Static site *content* (the blocks) lives in the `site` entity file via Store; only
the *live* sub-state (posts, presence, counters) lives in Yjs. This mirrors the
existing split (entity files on disk, sheet/overview live state in Yjs).

## B.4 The computer object on the sheet (new)

Requested: "a real computer object on the player sheet, with models and stats that
affect exploration." The sheet already has the netrunner's deck as `CS.netrunner`
(deck referenced by name into `DB.decks`; stats `_mu, speed, dataWall, codeGate`;
plus `programs[]`, `icon` persona, `interface`, `netAccessCode`, `heat`).

Plan: add a light **`CS.net`** block on the sheet (see D.5) holding:

- `computer` — the everyday machine for *non-runners*: a model chosen from a small
  new reference list `data/computers.json` (phones, pocket terminals, desktops,
  agents), each with `reach`, `power` (render/MU budget), `connection`
  (`landline` | `cellular` | `satellite`), `cost`. This is what gates exploration.
- The **cyberdeck (`CS.netrunner`) supersedes the computer** for netrunners: its
  `speed`/`_mu`/interface map onto reach/power, and it unlocks link-crossing a
  plain computer cannot. A character's *effective reach* = best of
  `CS.net.computer` and `CS.netrunner` deck.
- `address` — the delivery/residence address (district + building), reusing the
  Night City addressing convention (`district` + `building` + `floor`). Needed for
  purchases (B.5). New field; no address exists on the sheet today.
- `assets` — the netrunner's pre-established accesses (B.7).
- `accounts` visibility — money stays where it already is (`CS.lifestyle`, B.5).

`data/computers.json` is a new small DB following the flat-array + `cost`
convention of the other databases, browsable/sellable like any catalogue.

## B.5 Economy — reuse the sheet's money + shops, add delivery

The sheet already has everything except delivery:

- **Cash:** `CS.lifestyle.cash` (number, eb). Credchips: `CS.lifestyle.credchips`.
- **Bank accounts (unlimited):** `CS.lifestyle.accounts[]`, rich schema (balance,
  ledger, tier, bank ref). Debit/credit is direct on `acc.balance` +
  `_ledgerPush`, guarded by `_accCanDebit` (overdraft ceiling).
- **Purchase tray:** `pendPurchase(obj, kind, costOverride)` → `CS.pending`;
  `pendDebit()` settles selected lines from **cash or a chosen account**.
- **Shops already exist:** `shop` entity (`js/shop-editor.js` schema
  `{id,name,kind,url?,items:[{id,cat,name,price,qty,data}],notes,props}`) and the
  richer in-app shop in `app.js` (`shopBuy` → `applyPurchaseToSheet` →
  `shopWriteItem`, per-item `pitch`, `shop.link:{type,ref,url}`). Buying clones the
  DB row onto the sheet (`weapons`/`cyberware`/`vehicles`/`gear`) and appends a
  pending line.

What the Net adds (per decisions 4/12/13/14/15):

- **Online purchases require a bank account** (not cash). A shop-site's `buy`
  block routes through a new online settle path that debits an account
  (`_accCanDebit` + `acc.balance -=` + `_ledgerPush`), rather than the cash-or-tray
  choice.
- **Delivery.** A successful online purchase does **not** put the item on the sheet
  immediately; it creates a **delivery** record (D.6) with an ETA. The item lands
  in inventory when the delivery completes (driven by the in-game clock —
  `CS.lifestyle.month` already exists as a time counter; a finer per-session clock
  can be used). Delivery requires `CS.net.address`.
- **Delivery time is a shop attribute** (GM-set per shop) and a competitive axis
  between sites; it is also a play surface — a package can be delayed, intercepted,
  or seized (narrated by the GM as a scene in v1; systemic later).
- **Pricing:** base = catalogue `cost` with a per-shop markup/discount (already
  editable per line). **Fix the pricer** to fall back through
  `r.cost ?? r.bookcost ?? r.bookPrice ?? r.type?.cost` so vehicles and decks
  price correctly. Shops sell DB items or **custom entries** (`prep().customDb`,
  `_custom:true`), with optional per-line description (`pitch`; add the field to
  the newer `shop-editor.js` item schema for parity).
- **Traffic & ads (v2).** Each site carries a simulated `traffic` value (from reach
  × quality × inbound links × ad spend) with a GM override (a way to reward a
  player). A site can offer **ad slots** at a per-view rate (e.g. 0.001 eb);
  advertisers pay per view, revenue accrues to the host by traffic. Ad revenue is
  auto-credited to the owner's account. (Click-fraud dropped for now.)

## B.6 The Net map & browser — reuse the map stack

Navigation is a browser that unifies three things:

- **Address bar** — `grid://<region>/<site>` direct access. The address resolves to
  a `site` via `meta.idIndex` / a name index.
- **Map homepage** — the spatial view. Reuse the existing map engine: the Night
  City **6×3 district grid** (A1–C6, already named in `nightcity.html`'s `DATA`)
  plus the national/global **Leaflet** surface (the world/NA tiled map is already
  the root surface). Net regions = these existing units. A site is placed by its
  subject's district (or by broadcast tier for wide sites). The player's **reach**
  filters which site-nodes render; better computer = more nodes. The `datafort`
  location type already exists in `nightcity.js`'s `TYPES`, and the
  `linkedMapId` / `_createLinkedMap` nesting mechanism is exactly how we drill from
  a physical location into its Net presence.
- **Directory / search** — reuse the shell search/palette (`AppNav`) over the
  `site` type; categories from the site's `kind`.

The Net map is a **surface/view inside the `web` section**, not a replacement for
the physical map — but it can cross-link to it (a datafort node ↔ its physical
location) through Links.

## B.7 Netrunner assets & pre-established accesses (new sheet section)

A formal **"Net access & assets"** section on the netrunner sheet (`CS.net.assets`,
D.7). At character creation the netrunner spends a budget (scaled by level /
Lifepath, GM-approved) on persistent accesses drawn from a catalogue + free slots:

- live read of a site's registry/logs, a backdoor on a host, a forged credential,
  a tap on a comms channel, a planted program, a favour/debt from a host…

Accesses can grant **reach** to a site normally out of range (A.4), can be gained
or lost in play, and are **discoverable** (an access can be found and burned). They
turn the netrunner's backstory into live game state.

## B.8 Netwatch — heat + faction + content

Netwatch is a weather system, not a wall (decisions 18–21):

- **Heat** is a numeric notoriety carried **per character and per site** (both
  accrue and can be laundered).
- **Semi-automation:** heat thresholds *suggest* GM events; they never auto-punish.
- **Consequence ladder** (GM-triggered, defined but manual): patrol → investigate →
  pull the site → trace → physical response. A legit **host** may auto-flag illegal
  content, raising heat and inviting Netwatch (B.5/host legality).
- **Content, not just cops:** Netwatch has its own hackable sites (most-wanted —
  *are you on it?*, recruitment, propaganda), and its presence is asymmetric (strong
  in corp/Euro-style hosts and districts, weak on shady hosts / Wilderspace). The
  GM's monitoring of insecure chat apps (v2) is Netwatch surveillance made concrete.

## B.9 AIs, comms, cast (later phases)

- **AIs (v2):** a placeable actor (its own light entity or an `ai-host` block),
  dropped onto a site by drag-and-drop. Dialogue = a simple authored tree +
  **live GM override** (the GM types as the AI when it derails — the jank is the
  charm). Uncanny by near-miss; canon types (DHC/SAD/Human-AI-Soulkiller/CPP/
  Rogue/Transcendental).
- **Comms dock (v2):** a persistent phone/micro-deck at the bottom of the player
  shell (the Net always in pocket, via cellular). Chat apps are choosable clients
  with trade-offs (security/monitorable, reach, history, anonymity, cost, payload).
  Payloads: text, link (site), fiche/entity, money, program. The GM sees all
  traffic tool-side; a "secure" app is *marked non-interceptable in fiction*.
  Money transfer routes through the accounts system (B.5).
- **Cast (v2/v3):** cast a runner's live view to the table (Yjs awareness +
  the `overview`/`castReveal` mechanism already exists). Needs a clean
  "presentation" layout distinct from the runner's working UI. Spectators watch and
  react in chat; they do not act.

# PART C — Phased plan

Each phase is shippable on its own. The recurring design spine: **every tool the
player picks (computer, host, chat app, bank account, credential) is a choice with
trade-offs — and each is simultaneously a tool and an attack surface.**

## v1 — The Surface, end to end (the flashlight)

Goal: players autonomously browse a living Net of GM- and player-made sites, buy
things, and it all reaches the sheet. Netrunning is *not* in v1.

Scope:

1. **Module skeleton.** Enable the `web` section; `js/app-web.js` /
   `window.WebSection`; wire `renderPane`; `site` (and host-as-site) added to
   `Store.TYPES` + `createHub.mjs` `DOC_TYPES`; Links to subject entities.
2. **Site model + renderer.** JSON block document; a block registry
   (`{fields, render, kind}` per type). **Core ~12 blocks:** `header`, `text`,
   `image`, `list`, `link`, `board`, `compose`, `presence`, `item`, `price`,
   `form`, `gate`. Plus a starter `theme` + a first slice of the decor/old-net kit.
3. **Block editor (GM).** Data-model-first: edit the `blocks[]` array (palette add,
   drag-reorder, auto-generated field forms, live preview). The `gridmark` text
   form is an optional later serialization, not required.
4. **Style tool v1.** Theme presets + overrides (colour, background image, type,
   accent). Background/decorative images are first-class (drag-and-drop, plus images
   pulled from DATA fiches). A starter old-net kit (marquee, visitor counter,
   dividers, badges).
5. **Browser / explorer.** Address bar (`grid://`), map homepage (reuse district
   grid + Leaflet, reach-filtered), directory/search.
6. **Reach & computer.** `data/computers.json`; `CS.net.computer`; effective reach =
   best of computer / `CS.netrunner`. Sites carry `broadcast`. Map renders only
   what's reachable.
7. **Hosts.** Host = site + `hostConfig` (price, legality tolerance, Netwatch
   cooperation, reliability, reach granted, anonymity). Player sites reference a
   host. A basic host registry (directory of host-sites).
8. **Economy hookup.** Online `buy` debits a bank account (reuse `accounts` +
   `_accCanDebit` + `_ledgerPush`); creates a **delivery** (ETA, address); item
   lands on the sheet on completion. Fix the pricer fallback. Per-shop delivery
   time. Interception is GM-narrated in v1.
9. **Live blocks.** `board` + `presence` on the Yjs `net` map (players post, see
   who's online). Player autonomy: read-only Store + Yjs live.

Explicitly *not* in v1: authenticated corpo platforms, comms dock, Netwatch system,
AIs, traffic/ads economy, datafort/netrun, MicroNet.

## v2 — The living Net

Goal: depth, social life, and the middle tier. Everything that makes the surface
feel inhabited and risky, still without full netrunning.

Scope:

1. **Authenticated platforms (tier 2).** `login`/`auth`, `session`, per-block
   `access` scoping (public/employee/clearance:N/client/admin), family L blocks
   (dashboard, orgchart, webmail, docs by clearance, hr, badge/access-control,
   facilities, security-ops, finance, approvals, helpdesk, client-extranet, audit).
   Credentials as items/assets; the "one site, many faces by login" mechanic.
2. **Comms dock + chat apps.** Persistent phone/micro-deck; choosable chat clients
   with trade-offs; payloads text/link/fiche/money/program; GM monitoring; secure
   vs insecure apps.
3. **Traffic & ads economy.** Simulated traffic (+ GM override); ad slots at a
   per-view rate; auto-credited revenue.
4. **Netwatch system.** Heat (per char + per site), semi-automation thresholds, the
   consequence ladder, Netwatch's own hackable sites, host legality auto-flagging.
5. **AIs.** Placeable actors, drag-and-drop onto sites, authored dialogue tree +
   live GM override.
6. **Block breadth.** Add as many blocks as possible from families A–M (commerce,
   media, social, navigation, décor/old-net full kit, weird/diegetic). Style tool
   depth (full decor layer).
7. **Reviews/social, classifieds, subscriptions**, and the richer economy items.
8. *(Maybe)* first cast of a "site view" to the table.

## v3 — The deep end (netrunning)

Goal: the infrastructure layer and the netrunner's creative toolkit. This is the
big module; it dovetails with the physical map and PvP.

Scope:

1. **Datafort (tier 3).** `datafort` entity under a site (nodes, data walls, code
   gates, ICE, files by type). Discovery + entry from the Net map (never a button
   on the site). The two seams made real: forge a credential from a cracked
   datafort → unlock the tier-2 platform.
2. **Netrun UI + resolver.** Turn-based run (canon: 1 action/round, initiative,
   STR rolls). Uses the program list (`cp2020programs.json`) and the deck
   (`CS.netrunner`).
3. **Casting runs.** The clean presentation layout; cast to table via
   Yjs/overview; spectators in chat.
4. **Creation tools.** Program writer (Programming 101), cyberdeck building, gadget
   & trap coding, and **player-built dataforts** (Fast Fortress Construction) to
   defend their own business → emergent PvP (another runner attacks a player's
   datafort).
5. **Generators.** Datafort generator (guided + random "Fast Fortress"), program
   generator, tied into the databases.
6. **MicroNet** — only once the **Item creator** exists (see E.1); an item's Net
   presence is its MicroNet (the 4-task object hack).

# PART D — Schema appendix

Draft shapes; ids via the existing `uid`/`_bankUid` helpers. All entity files
follow the Store convention (`id` inside the JSON, `props`, refs via Links).

## D.1 `site`
```js
{
  id, name,
  kind,                 // 'venue'|'bbs'|'media'|'corp'|'gov'|'shop'|'person'|'host'|'blackmarket'|...
  address,              // { region, slug }  → grid://region/slug
  subject,              // ref {type,id} to location|org|shop|npc (via Links too)
  hostId,               // ref to the host-site this is published through (player sites)
  broadcast,            // 'local'|'district'|'citywide'|'national'|'global'
  region,               // district id (A1..C6) | national | global
  theme,                // { preset, overrides:{bg, accent, type, decor:[...] } }
  blocks: [ Block ],    // ordered document (see D.2)
  state,                // { online:true, hours?, poweredBy? }  → the "lamp"
  unlisted,             // bool (reachable by address, hidden from directory)
  hostConfig?,          // present only when kind==='host' (see D.3)
  datafortId?,          // GM-only backend link to tier-3 infra (v3)
  props                 // { public, contact, ... }  → player visibility
}
```

## D.2 `Block`
```js
{
  id, type,             // registry key: header|text|image|list|link|board|compose|
                        //   presence|item|price|form|gate|... (families A–M)
  access,               // 'public'|'employee'|'client'|'admin'|{clearance:N}  (default 'public')
  ...fields             // per-type fields, described by the registry's field schema
}
// Live blocks (board/chat/presence/traffic) keep only config here; their live
// data lives in the Yjs `net` map keyed by siteId.
```

## D.3 `hostConfig` (on a host-site)
```js
{
  price,                // eb / period to host here
  legality,             // 0..3  how much illegal content it tolerates
  netwatchCoop,         // 0..3  keeps logs / rats you out
  reliability,          // 0..3  uptime
  reachGranted,         // max broadcast tier it can give a client site
  anonymity,            // 0..3  how well it hides the owner
  serverAddress,        // the real address (recon reward; hidden until hacked)
}
```

## D.4 `data/computers.json` (new reference DB, flat array)
```js
{
  name, class,          // 'phone'|'pocket'|'desktop'|'agent'|'cyberdeck'
  reach,                // 0=local, 1=district, 2=citywide, 3=national, 4=global
  power,                // render/MU budget (site richness it can display)
  connection,           // 'landline'|'cellular'|'satellite'
  cost,                 // eb
  source:{book,page}
}
```

## D.5 `CS.net` (new sheet block)
```js
CS.net = {
  computer,             // { model, ...resolved stats }  (non-runner machine)
  address,              // { district, building, floor }  (delivery/residence)
  deliveries: [],       // Delivery[]  (see D.6)
  assets: [],           // NetAsset[]  (see D.7)
  bookmarks: [],        // saved grid:// addresses
  history: []           // recent addresses
}
// Netrunner deck stays where it is: CS.netrunner. Effective reach =
// max(reach(CS.net.computer), reach(CS.netrunner deck)).
```

## D.6 `Delivery`
```js
{ id, itemRef, name, cat, price, fromShopId, address,
  placedAt, eta, status:'in_transit'|'delivered'|'seized'|'delayed', accountId }
// On 'delivered': clone the item onto the sheet array (weapons|cyberware|vehicles|gear),
// same as the existing shopWriteItem path.
```

## D.7 `NetAsset` (pre-established access)
```js
{ id, kind,             // 'registry_read'|'host_backdoor'|'forged_cred'|'comms_tap'|
                        //   'planted_program'|'host_favour'|'custom'
  target,               // ref {type,id} (site/host/...) or free text
  grantsReach,          // bool  (can reach an otherwise out-of-range site)
  discoverable,         // bool  (can be found & burned)
  status:'active'|'burned'|'expired', notes }
```

## D.8 Yjs `net` map (live sub-state)
```js
net[siteId] = {
  board: [ {id, handle, ts, tag, body, signal, replies:[...] } ],
  presence: { memberId: {handle, since} },
  traffic: { views, adImpressions }
}
```

# PART E — Deferred & open

- **E.1 MicroNet (item hacking).** Deferred until the **Item creator** exists (a
  broader, mostly-techie tool not yet designed). An item's Net presence is its
  MicroNet (the 4-task object hack: Connect → Navigate → Control → Alter). Reserve
  the seam (`item → MicroNet`); do not build until Items land. Target: v3.
- **E.2 Computer vs deck unification.** Decision to confirm during v1: keep
  `CS.net.computer` (non-runner) and `CS.netrunner` (deck) separate with an
  effective-reach max, vs. a single unified device list. Recommendation: separate,
  unify the *reach* computation only.
- **E.3 Pricer fix** (`shopPrice`) must land with the first shop-site work:
  fall back `cost ?? bookcost ?? bookPrice ?? type.cost`.
- **E.4 Click fraud** — dropped for now (may return as a dark-pattern action).
- **E.5 Region model beyond Night City.** v1 uses NC districts + national/global
  tiers as-is ("everything you capture at Night City"). A fuller region graph
  (other cities) can come later without schema change (`region` is already a free
  id).
- **E.6 Remote play.** LAN only, by design. The Yjs/WebSocket layer would allow
  remote later, but it is out of scope.
